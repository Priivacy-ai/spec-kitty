## Subtask T035 — ADR-A: Global `~/.kittify/` as Machine-Level Runtime

**Filename:** `2026-04-08-1-global-kittify-machine-level-runtime.md`

**Core decision:** `spec-kitty init` bootstraps a single global runtime at `~/.kittify/`. Per-project `.kittify/` directories are thin overlays, not full copies.

**What to cover:**
- **Context:** Before this decision, init copied full template sets to each project directory. This meant each project had its own copy of missions, scripts, and templates — duplicating data, making upgrades inconsistent, and blurring the line between machine setup and project setup.
- **Decision:** `~/.kittify/` is the canonical location for missions, globally-installed skills, and agent configuration. Running `spec-kitty init` bootstraps this global directory. Per-project `.kittify/` contains only project-specific state (charter, feature specs, local overrides).
- **What belongs where table:** missions → global; charter → per-project; agent config → global; feature specs → per-project; skills → global canonical, per-project symlinks.
- **Upgrade behavior:** Re-running init updates the global runtime without touching projects.
- **Error surfacing:** `ensure_runtime()` failures must not be silently swallowed (contrast with pre-076 behavior).
- **Alternatives:** (1) Per-project full copy (status quo pre-076) — rejected: data duplication, inconsistent upgrades; (2) Read-only global + writable project overlay — considered but adds complexity without benefit at this stage.
- **Cross-references:** ADR-C (skills), ADR-D (charter), spec.md FR-003.

## Subtask T036 — ADR-B: Package-Bundled Templates as Sole Source

**Filename:** `2026-04-08-2-package-bundled-templates-sole-source.md`

**Core decision:** Template assets are always sourced from the installed spec-kitty-cli package. Remote GitHub tarball download and the `--template-root` CLI flag are removed.

**What to cover:**
- **Context:** init had three template modes: local repo (dev override via `--template-root` or env var), package-bundled (default), and remote GitHub download. The remote mode was originally designed for a bootstrap-without-install scenario that no longer applies. The `--template-root` CLI flag had no end-user use case.
- **Decision:** Package-bundled templates are the unconditional default. The `SPEC_KITTY_TEMPLATE_ROOT` environment variable remains available for maintainers testing template changes locally. `--template-root` CLI flag is removed. `github_client.py` is deleted.
- **Consequences:** No network access required during init. `--debug`, `--skip-tls`, `--github-token` flags (which were only used for the remote download path) are also removed.
- **Alternatives:** (1) Keep remote mode behind a hidden flag — rejected: adds maintenance burden, no known use case; (2) Keep `--template-root` as a hidden flag — rejected: env var is sufficient for the only use case (maintainer dev testing).
- **Cross-references:** ADR-A (global runtime), spec.md for the removed flags table.

## Subtask T037 — ADR-C: Global Skill Installation with Per-Project Symlinks

**Filename:** `2026-04-08-3-global-skill-installation-per-project-symlinks.md`

**Core decision:** Skills are installed canonically to `~/.kittify/agent-skills/{agent}/`. Per-project agent directories reference skills via symlinks (copy fallback).

**What to cover:**
- **Context:** Previously, skills were installed per-project during `spec-kitty init`. Each project got its own copy of every skill for every agent. This meant upgrades required re-running init on every project, and skills were duplicated across the filesystem.
- **Decision:** Skills are global. `spec-kitty init` installs skills to `~/.kittify/agent-skills/{agent}/`. Per-project setup (a future concern) creates symlinks into the global installation. Copy fallback for filesystems that don't support symlinks.
- **Known gap:** The current implementation installs per-project wiring during init (a known limitation). This ADR documents the intended architecture; per-project wiring will be moved to a dedicated setup flow in a future feature.
- **Alternatives:** (1) Per-project full copy (status quo) — rejected: upgrade burden, disk waste; (2) Symlink only (no copy fallback) — rejected: symlinks not universally available (network shares, some Windows configurations).
- **Cross-references:** ADR-A, spec.md FR-007.

## Subtask T038 — ADR-D: Charter and Doctrine Are Not Init-Time Concerns

**Filename:** `2026-04-08-4-charter-doctrine-not-init-time.md`

**Core decision:** All doctrine/charter interview code is removed from `spec-kitty init`. Charter is per-project governance, authored via `/spec-kitty.charter`.

**What to cover:**
- **Context:** `spec-kitty init` previously ran a full charter/doctrine interview: `_run_doctrine_stack_init()`, `_run_inline_interview()`, `_apply_doctrine_defaults()`. This ran before the user had a project, which is logically wrong — the charter describes a specific project's governance, and no project exists yet at machine-init time.
- **Why it accumulated:** Historical artifact. Charter was added to init when spec-kitty was a per-project tool, before the global runtime model existed.
- **Decision:** Init is machine setup. Charter is project governance. These are distinct concerns. Charter is authored via `/spec-kitty.charter` after `/spec-kitty.specify` creates the project.
- **Impact:** `test_init_doctrine.py` deleted. Three large function definitions removed from init.py.
- **Alternatives:** (1) Keep charter in init, make it optional — rejected: init would still prompt for project-specific decisions before the project exists; (2) Move charter to a `spec-kitty setup` per-project command — this is the intended future direction; for now, `/spec-kitty.charter` is the surface.
- **Cross-references:** ADR-A, spec.md FR-015.

## Subtask T039 — ADR-E: Shim Generation Supersedes Script-Type Dispatch

**Filename:** `2026-04-08-5-shim-generation-supersedes-script-dispatch.md`

**Core decision:** The `.kittify/scripts/bash/` and `.kittify/scripts/powershell/` directories, `--script` flag, and script-type selection logic are removed. Agent command dispatch uses shim generation exclusively.

**What to cover:**
- **Context:** Early versions of spec-kitty dispatched agent commands via shell scripts (bash on Unix, PowerShell on Windows). init asked the user which script type to use, copied the appropriate variant to `.kittify/scripts/`, and configured agents to call these scripts.
- **What replaced it:** `shims/generator.py` — writes thin markdown files directly into agent command directories (`.claude/commands/`, `.codex/prompts/`, etc.). These are native to the agent's command format and require no shell dispatch.
- **Current state at removal:** No `.sh` or `.ps1` files exist in the package. The `scripts/bash/` and `scripts/powershell/` directories are copied during init but never invoked — pure dead code.
- **Why safe to remove:** `shims/generator.py` is the active dispatch mechanism. The script files were cleaned up on init completion anyway.
- **Alternatives:** (1) Keep scripts as fallback for agents that don't support markdown commands — rejected: all 13 supported agents use markdown/text command files; (2) Deprecate with a warning before removal — rejected: the code is already dead; no users depend on it.
- **Cross-references:** `src/specify_cli/shims/generator.py`, spec.md "Code Paths to Remove".

## Subtask T040 — ADR-F: Global Agent Commands Supersede Per-Project Copies

**Filename:** `2026-04-08-6-global-agent-commands-supersede-project-copies.md`

**Core decision:** Global agent command files (`~/.claude/commands/spec-kitty.*.md`, etc.) are canonical. Per-project copies are removed by migration when global equivalents are confirmed present.

**What to cover:**
- **Context:** When spec-kitty was a per-project tool, it wrote slash command files into `.claude/commands/`, `.codex/prompts/`, etc. within each project. After the global runtime was introduced, commands were also installed globally. Projects that upgraded to the global model now have both local and global command files — causing users to see duplicate slash commands in their AI tools.
- **Migration 3.1.2 (existing):** `m_3_1_2_globalize_commands.py` removes local command files unconditionally — no check for global presence. On a machine without `~/.kittify/`, this deletes all command files with no recourse.
- **Migration 3.2.2 (this feature):** `m_3_2_2_safe_globalize_commands.py` implements four safety invariants before any deletion (global runtime present, global agent commands present, version marker check, per-agent skip on failure).
- **Why not edit m_3_1_2:** The migration runner tracks by `migration_id`. In-place edits would never re-execute on projects that already ran 3.1.2.
- **Version marker as fingerprint:** `<!-- spec-kitty-command-version: X.Y.Z -->` on line 1 identifies spec-kitty-generated files. User-authored files without this marker are never touched.
- **Alternatives:** (1) Harden m_3_1_2 in-place — rejected: migration ID tracking means it silently skips existing projects; (2) Force re-run via a flag — rejected: adds complexity; new ID is the correct mechanism.
- **Cross-references:** spec.md FR-021–FR-023, `m_3_1_2_globalize_commands.py`, `shims/generator.py:101`.

## Subtask T041 — ADR-G: Preferred Agent Roles Removed as Unused Concept

**Filename:** `2026-04-08-7-preferred-agent-roles-removed-unused.md`

**Core decision:** `AgentSelectionConfig`, `select_implementer()`, `select_reviewer()`, and all `selection` serialization are removed from the codebase. The preference system was collecting data no runtime code ever consumed.

**What to cover:**
- **Context:** `AgentSelectionConfig` stored `preferred_implementer` and `preferred_reviewer`. These were set during `spec-kitty init`, serialized to `config.yaml`, and loaded by `load_agent_config()`. The methods `select_implementer()` and `select_reviewer()` were defined to route work to the preferred agents.
- **Discovery:** Code audit for this feature (076) revealed that `select_implementer()` and `select_reviewer()` are defined in `agent_config.py` and called **nowhere in the entire codebase**. Not from any CLI command, migration, or runtime code. The `agent config` management CLI does not expose the fields. The data was written and read but never acted upon.
- **Decision:** Remove the entire preference layer. `AgentSelectionConfig` class deleted. Both methods deleted. `selection` block removed from `save_agent_config()` and `load_agent_config()`. Cleanup migration (3.2.1) strips orphaned keys from existing `config.yaml` files.
- **Pre-existing bug fixed:** `load_agent_config()` read only from `agents` key, but migration 2.0.1 renamed it to `tools`. This silent data loss bug was discovered during the same audit and fixed in WP01 (try `agents` → fallback to `tools`).
- **Alternatives:** (1) Implement the routing feature — rejected: no specification exists for how routing should work; implementing an undocumented feature would be speculative; (2) Deprecate with a warning — rejected: the methods are never called; there are no users to warn.
- **Cross-references:** spec.md FR-017–FR-020, ADR-6 (config-driven agent management), `core/agent_config.py`.
