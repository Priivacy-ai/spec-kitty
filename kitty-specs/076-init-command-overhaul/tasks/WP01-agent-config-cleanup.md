# WP01 — Agent Config Data Model Cleanup

## Objective

Remove the entirely unused `AgentSelectionConfig` class, `select_implementer()`, and `select_reviewer()` from `src/specify_cli/core/agent_config.py`, along with all serialization/deserialization of the `selection` block. Also fix the pre-existing silent bug where `load_agent_config()` reads from `agents` key but migration `m_2_0_1_tool_config_key_rename.py` renames it to `tools`.

**Why this WP comes first:** `init.py` (WP02) imports `AgentSelectionConfig` and uses it at line 975. WP02 cannot pass mypy until this WP removes the class.

## Branch Strategy

- **Planning base:** `main`
- **Merge target:** `main`
- **Implementation command:** `spec-kitty implement WP01`
- Execution worktree: resolved from `lanes.json` after `finalize-tasks`

## Context

`src/specify_cli/core/agent_config.py` (~240 lines) contains:

- `AgentSelectionConfig` dataclass (lines 28–37): stores `preferred_implementer: str | None` and `preferred_reviewer: str | None`
- `select_implementer()` method on `AgentConfig` (lines 57–72): looks up `preferred_implementer` in candidates, falls back to first available — **never called anywhere in the codebase**
- `select_reviewer()` method on `AgentConfig` (lines 75–99): mirror for review — **never called anywhere in the codebase**
- `selection: AgentSelectionConfig` field on `AgentConfig` dataclass
- `load_agent_config()` reads `selection_data = agents_data.get("selection", {})` (lines 157–162) and constructs `AgentSelectionConfig` from it
- `save_agent_config()` writes `"selection": {"preferred_implementer": ..., "preferred_reviewer": ...}` (lines 194–196)
- `AgentSelectionConfig` is in `__all__` (line ~241)

**Pre-existing bug (T006):** `m_2_0_1_tool_config_key_rename.py` renames the root key `agents` → `tools` in config.yaml. But `load_agent_config()` only reads from `data.get("agents", {})`. On projects that ran 2.0.1, the agent config silently returns empty defaults. Fix by reading `data.get("agents") or data.get("tools", {})`.

## Subtask T006 — Fix `load_agent_config()` for `agents → tools` Key

**Purpose:** Fix the pre-existing silent bug where projects that ran migration 2.0.1 (`agents → tools` rename) would get empty agent config on subsequent `load_agent_config()` calls.

**Steps:**

1. In `load_agent_config()`, find the line:
   ```python
   agents_data = data.get("agents", {})
   ```
2. Replace with:
   ```python
   agents_data = data.get("agents") or data.get("tools") or {}
   ```
   This tries `agents` first (the common case), then `tools` (for projects that ran migration 2.0.1), then empty dict.

3. Do the same for the `auto_commit_raw` parsing which also reads from `agents_data` — confirm it's already reading from the variable (not re-reading `data["agents"]` directly).

**Validation:**
- A config file with `agents: {available: [claude]}` → returns `AgentConfig(available=["claude"])`
- A config file with `tools: {available: [opencode]}` → returns `AgentConfig(available=["opencode"])`
- A config file with neither key → returns `AgentConfig()` (empty)
