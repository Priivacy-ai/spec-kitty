# WP06 — Codex Legacy Migration

## Objective

Deliver a one-shot upgrade migration that moves every existing Codex user from `.codex/prompts/spec-kitty.*.md` to `.agents/skills/spec-kitty.<command>/SKILL.md` with zero manual steps. Preserve user-edited files with a clear notice. Never touch third-party files. Produce a working Codex integration after a single `spec-kitty upgrade` invocation.

## Context

Before this mission, Spec Kitty rendered command files for Codex into `.codex/prompts/`. WP04 removed `codex` from `AGENT_COMMAND_CONFIG` and WP03 provides the installer that writes the new skill packages. This migration closes the loop by cleaning up the legacy artifacts on existing projects.

The classification algorithm (from `data-model.md` §LegacyCodexPrompt):

- File's current SHA-256 matches the known previous-release hash → **owned_unedited** → delete after new skill is installed.
- File's SHA-256 does not match but filename is `spec-kitty.*.md` → **owned_edited** → preserve, print notice.
- Filename is not `spec-kitty.*.md` → **third_party** → ignore completely.

The migration must be no-op when `codex` is not in `agents.available` — a Vibe-only project should not trigger this migration.

### Subtask T030 — Implement `apply()`

Behavior:

```python
def apply(self, project_path: Path, dry_run: bool = False) -> None:
    config = load_agent_config(project_path)
    if "codex" not in config.available:
        return  # no-op per T032

    prompts_dir = project_path / ".codex" / "prompts"
    if not prompts_dir.exists():
        # Project may never have initialized codex on disk. Still run the installer
        # so the new skills are present.
        _install_skills(project_path, dry_run)
        return

    classified = self._classify(prompts_dir)

    if not dry_run:
        _install_skills(project_path, dry_run=False)

    owned_unedited = [p for p in classified if p.status == "owned_unedited"]
    owned_edited   = [p for p in classified if p.status == "owned_edited"]

    for p in owned_unedited:
        if not dry_run:
            p.path.unlink()

    if owned_edited and not dry_run:
        _print_preservation_notice(owned_edited)

    # Remove the prompts dir only if it is empty. Never touch .codex itself.
    if not dry_run and prompts_dir.exists() and not any(prompts_dir.iterdir()):
        prompts_dir.rmdir()

def _install_skills(project_path: Path, dry_run: bool) -> None:
    if dry_run:
        return
    from specify_cli.skills import command_installer
    command_installer.install(project_path, "codex")
```

`_classify(prompts_dir)` enumerates `*.md` files under `prompts_dir`, hashes each, and compares against `LEGACY_CODEX_HASHES`. Files whose name is not in the dict are classified as `third_party` and ignored. Files whose current hash matches the known hash are `owned_unedited`; otherwise `owned_edited`.

`_print_preservation_notice(edited)` prints (via Rich or plain stderr — follow existing migration style) a user-facing notice:

```
Preserved user-edited files in .codex/prompts/:
  • .codex/prompts/spec-kitty.specify.md
  • .codex/prompts/spec-kitty.plan.md

Your Codex integration now reads from .agents/skills/; review and port
your edits if still needed.
```

## Risks

- **Hash drift from renderer churn.** If the pre-3.2 renderer is still evolving on `main` at mission start, the known hashes could become stale. Pin the hash-regeneration script to a specific commit of the pre-mission code and record that commit in a comment at the top of `_legacy_codex_hashes.py`.
- **User's `.codex/prompts/` contains nested directories.** The classifier must only look at `*.md` files directly inside `prompts/`, not recurse (pre-3.2 renderer never produced nested layout). Document this constraint.
- **`.codex/` directory removal.** The migration removes only `.codex/prompts/` when empty. Never touch `.codex/` itself — it may contain unrelated Codex config.
- **Fixture realism.** If the "known hashes" table is generated from a reference template, a future template drift could break every pre-3.2 project even for unedited files. Treat the hash table as versioned artifact; changes to it require a migration bump.
