## Objective

Add a discrete GitHub Actions workflow file in this repo (`Priivacy-ai/spec-kitty`) that runs the canonical-registry drift-detector test (`tests/sync/test_diagnose.py::TestCanonicalRegistryRecognition`) on every PR. The workflow exposes a single, stable job named `drift-detector` that a repo admin can register as a required check on `main`.

## Context

- The drift-detector test exists today at `tests/sync/test_diagnose.py:417` (class `TestCanonicalRegistryRecognition`).
- It currently runs as one of many slices inside `ci-quality.yml`, where the job name varies per slice (e.g. `test (3.11)`). Branch protection cannot register a matrix-varying name reliably.
- A discrete workflow file gives us one stable job name.
- Filename collision context: sibling open PR #1250 (canonical-producer-lint) uses `canonical-producer-lint.yml`. Our file MUST be named `drift-detector.yml`.

## Subtasks

### T001: Author `.github/workflows/drift-detector.yml`

**Purpose**: Create a workflow file that triggers on PRs against main and runs the named drift-detector test in a single job.

**Steps**:
1. Create the file `.github/workflows/drift-detector.yml` with this content:

```yaml
# .github/workflows/drift-detector.yml
#
# Identity-Boundary CI Gate — spec-kitty repo
#
# Tracker: https://github.com/Priivacy-ai/spec-kitty/issues/1247
# Closes the drift-class hole that took rc14 -> rc22 to peel: a PR that
# changes the canonical registry without updating consumer recognition
# code must fail fast at PR time, not at the next on-demand canary run.
#
# The single job named `drift-detector` is the contract this workflow
# offers to branch protection. Do not rename without coordinating with
# the admin who registered the required check.
#
# SHA-bump procedure: see README.md > "Identity-Boundary CI Gate" for
# how to coordinate canonical-registry contract changes with downstream
# consumers.

name: drift-detector

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

concurrency:
  group: drift-detector-${{ github.ref }}
  cancel-in-progress: true

jobs:
  drift-detector:
    name: drift-detector
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install uv
        uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true

      - name: Install dependencies (frozen)
        run: uv sync --frozen --extra dev

      - name: Run canonical-registry drift detector
        run: uv run pytest tests/sync/test_diagnose.py::TestCanonicalRegistryRecognition -v
```

2. Verify the job's top-level `name:` and the job-id `drift-detector:` AND the job's `name:` field are all `drift-detector`. GitHub Actions reports the job's `name:` to the PR checks list; branch protection registers against that exact string.

## Edge cases / risks

- **Job name drift**: A future PR renames the job; branch-protection registration breaks silently. Mitigation: README section explicitly names the check string; reviewer-renata watches for renames.
- **`uv sync --frozen --extra dev` fails in CI but passes locally**: usually means the lockfile is out of date for the dev extras. Out of scope for this WP; if it bites, treat as a regression and split into a follow-up.
- **Test becomes flaky**: not expected; the test is deterministic. If it bites post-merge, file an issue, do not change the workflow.
