2026-07-15 — Finding (WP06 T027, CORRECTED — #2647 DOES reproduce; root cause is #2453, out of WP06 scope). The real taint is NOT in `tasks_move_task.py`'s reads (all three of `_read_transactional_wp_lane`, `locate_work_package`, and `resolve_action_context`/`_assemble_core_fragments` are confirmed CWD-invariant — they govern the DECISION half of `move-task` correctly). It is in the WRITE half: `emit_status_transition_transactional` → `BookkeepingTransaction.acquire` → `_acquire_locked` (`src/specify_cli/coordination/transaction.py`) independently RE-DERIVES `from_lane` at commit time via `_derive_from_lane(txn.feature_dir, wp_id)`, where `txn.feature_dir` comes from `_resolve_legacy_lane_destination` (transaction.py:279) whose FIRST line is `cwd = Path.cwd().resolve()`. `_is_legacy_mission` (transaction.py:200) classifies ANY mission whose `meta.json` lacks a `coordination_branch` key as "legacy" — true for every modern coordination-less `single_branch`/`lanes` mission (per #2218), not only genuinely pre-SSOT ones. For a lane worktree forked BEFORE later status transitions land on the primary/coordination branch, its own git-tracked local `kitty-specs/<slug>/status.events.jsonl` snapshot is stale, so the write-time re-read returns the WRONG (stale) `from_lane`, and `validate_transition` correctly refuses the resulting (wrong) edge — producing exactly "Illegal transition: planned -> for_review". Reproduced this LIVE on THIS mission's own WP06 (`kitty-specs/implement-loop-commit-hardening-01KXJ1ZX/status.events.jsonl` showed `in_progress` on the primary checkout; the lane-c worktree's own tracked copy still showed the `planned` bootstrap event from before the `claimed`→`in_progress` events were appended) — the mission's own move-task handoff hit this bug. Also reproduced in an isolated, controlled fixture: `tests/specify_cli/cli/commands/test_tasks_move_task_cwd.py::test_write_side_from_lane_rederivation_reproduces_2647` (real `meta.json` with `mission_id` + `topology: single_branch`, no `coordination_branch`; lane worktree forked at the `planned` fork point; primary advances to `in_progress` after fork) — RED with the identical error string, confirmed via direct debug run before marking `xfail(strict=True)`. **Critically, `BookkeepingTransaction.commit()`'s `CommitTarget(ref=self.destination_ref)` construction at this exact site is ALREADY a tracked, DEFERRED architectural-debt allow-list entry** in `tests/architectural/test_no_write_side_rederivation.py` (`_CHECKOUT_GRAMMAR_ALLOW_LIST_SEED`, rationale text: "tracked: #2453 - BookkeepingTransaction.commit()'s legacy-mission override (_resolve_legacy_lane_destination reads the lane worktree's HEAD) ... deferred to the #2453 sweep") — the project already knows this exact code is checkout-derived and has explicitly deferred fixing it, just not previously connected to #2647's user-visible symptom. A sibling, already-merged mission (`topology-aware-legacy-warning-01KWQ8WH`, #2351) fixed the misleading "legacy" WARNING TEXT for coordination-less modern missions but explicitly preserved this write-routing behavior (C-005: "that predicate [_is_legacy_mission] ... continues to drive worktree routing ... unchanged" — a deliberate scoping decision for THAT mission, not a permanent invariant). Decision: WP06 does NOT attempt to fix `coordination/transaction.py` — it is not in WP06's `owned_files` (`tasks_move_task.py` + its test only), it is a shared, heavily-relied-upon coordination primitive used by every status transition system-wide, and it carries its own architectural ratchet/allow-list contract gating changes to it. Converting WP06's test to a false "already fixed" regression guard (the prior, now-retracted entry) would have shipped a misleading GREEN over a live, reproducing bug. Instead: `test_write_side_from_lane_rederivation_reproduces_2647` is committed as `xfail(strict=True)`, referencing both #2647 and #2453, so (a) the bug is honestly characterized rather than hidden, (b) a future #2453 fix flips this test GREEN automatically (turning `xfail` into an unexpected-pass failure that FORCES the marker's removal), and (c) WP06 reports this as a blocker/scope-escalation for the operator rather than silently absorbing #2453's debt into a narrow WP. Alternatives considered: (a) fix `_resolve_legacy_lane_destination` to route modern coordination-less missions (real `mission_id` + stored `topology`) to `repo_root` + `target_branch` instead of `Path.cwd()`, reusing the SAME classification `_warrants_legacy_warning` already established — this is very likely the RIGHT eventual fix, but requires updating/removing the tracked #2453 allow-list entry, adding coverage for genuinely-legacy (pre-SSOT, no stored topology) missions to prove their CWD-based routing is preserved, and re-running the full `coordination/` + `status_transition` test suites for regressions — sized for its own WP/mission, not a scope-add to WP06; (b) work around it by re-syncing the lane worktree's local kitty-specs copy before every move-task call — rejected, papers over the structural bug and doesn't fix it for other agents/lanes; (c) use `--force` on the real WP06 handoff — rejected, `force` bypasses the edge check but the WRITE would still persist the WRONG `from_lane` ("planned") into the permanent event log, corrupting history; the safe workaround for completing THIS mission's own handoff is to run `move-task` from the PRIMARY repo root (matching the original #2647 bug report's own "succeeds from repo root" observation) rather than the lane-c worktree.
