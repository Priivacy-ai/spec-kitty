# Tracer — Tooling Friction

Append friction encountered during the mission; feeds the next mission.

## F-1 — Basing a mission on an unmerged PR branch is manual

The fix these issues patch exists only on PR #2732's `fix/2681-synthesized-drg-stale`, not on
`main`. There is no first-class "start a mission stacked on an open PR branch" flow — required
manual steps: reset the stale local branch to the fork PR head, create a dedicated mission branch
off it, verify `git merge-base --is-ancestor`. Worth a helper if stacked-mission work recurs.

## F-2 — `spec-kitty specify` defaults `target_branch` to the current branch

Creating a mission while checked out on a PR branch would commit `meta.json` onto that PR branch
(polluting it). Had to create a separate mission branch first. The commit message auto-generated by
mission-create still says "feature" ("Add meta for feature …") — a terminology-canon drift in the
tool's own commit text (not agent-authored).

## F-3 — Tracker origin-binding is a SaaS "pending-origin" flow, not provisioned for a fork contributor

`spec-kitty specify` reported `origin_binding.attempted=false`; binding an issue requires a
pre-created pending-origin (ticket-first SaaS flow). For a cross-fork contributor the canonical
traceability is the spec citing both issues + the final PR linking them (`Fixes #2758`,
`Fixes #2759`), so binding was not forced.

## F-5 — `spec-kitty implement` preflight fail-closes on a pre-existing invalid charter manifest

`spec-kitty implement`/`next`/status all run a charter preflight that fail-closes with
`synthesized_drg missing; run spec-kitty charter synthesize`. The repo's own
`.kittify/charter/synthesis-manifest.yaml` is INVALID (integrity mismatch, no `graph.yaml`) — a
**pre-existing** state shipped on origin/main from an ancient fixture run (`01KPE222`, adapter
`fixture v1.0.0`), unrelated to this mission. Running `charter synthesize` to clear it would modify
tracked charter files (`synthesis-manifest.yaml`, provenance/, generated graph) and pollute the
mission diff with unrelated repo-charter-repair. **Mitigation:** since this is `single_branch` (the
current branch is the workspace), implemented directly on `gk/2758-2759` with git + ATDD discipline,
bypassing the environmentally-blocked workspace command. Reported here per the Pre-existing Failure
Reporting Rule; noted for the PR body rather than opening upstream issue-noise from a fork contributor.

## F-6 — Pre-existing mypy `no-any-return` in `computer.py` (isolated-file artifact)

`uv run mypy src/specify_cli/charter_runtime/freshness/computer.py` in isolation reports 3
`no-any-return` errors (lines ~161/229/230). **Pre-existing** — the SAME errors are present on the
base `fix/2681-synthesized-drg-stale`, and this mission's only change to `computer.py` is the module
docstring (zero code lines). They are an isolated-single-file-mypy artifact: the `charter.*`
`follow_imports = "skip"` override (pyproject) erases cross-module return types to `Any` when the
file is checked alone; the full-repo mypy resolves them. This mission's NEW code (`bundle.py`,
`compiler.py`) is mypy-clean. Reported per the Pre-existing Failure Reporting Rule; noted for the PR
body; not fixed here (out of scope, not a regression).

## F-4 — Multiple parallel "list of bundle files" constants invite drift

`CANONICAL_MANIFEST.derived_files` (3), `BUNDLE_CONTENT_HASH_FILES` (4), and `computer._BUNDLE_FILES`
(4, a deliberate duplicate for import-direction reasons) serve different signals but look alike; the
squad flagged the risk of a 4th informal list. C-003 constrains the mission to edit only
`BUNDLE_CONTENT_HASH_FILES`. A single reconciled source-of-truth would reduce this class of error.
