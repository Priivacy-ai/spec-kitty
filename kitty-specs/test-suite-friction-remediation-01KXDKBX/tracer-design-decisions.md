## ‚≠ź STANDING CATALOG ‚ÄĒ ratchet / no-regression-pinning & behavioural-parity test suites (operator hypothesis, 2026-07-13)
**Operator's suspicion**: the ratchet / "no regression pinning" suites and the "behavioural parity"-type tests, **mass-added over the last few cycles, do more harm than good** ‚ÄĒ they are scaffold written to support a mission's development that becomes mostly noise/friction afterwards (the `tests-as-scaffold-not-friction` thesis, #2071). The CaaCS analysis already fingered the ratchets as the single largest maintenance tax.

**Obligation**: as each mission WP touches or observes one of these suites, append a row below. At mission close, produce a **keep / consolidate / retire verdict** per suite and file a follow-up mission/issue for the net-negative ones. This mission does NOT remediate them wholesale (unbounded) ‚ÄĒ it CATALOGS with evidence so the follow-up is data-driven, not vibes.

**Discriminator** (from research.md / CaaCS): a pinning test earns its keep ONLY if it pins a *behavioural / negative invariant* (import ban, layer ban, security boundary). If it pins *positive code shape* (a symbol lives at a path, a count, a call reads a string), or it co-changes with src on every refactor (ratio‚Üí1.0), it is scaffold-friction ‚Üí convert-to-behavioural or retire.
