# WP05 — CLI decision subgroup (typer)

## Objective

Expose the Decision Moment Python API (WP03) and verifier (WP04) as a first-class CLI
subgroup: `spec-kitty agent decision {open, resolve, defer, cancel, verify}`. Each
subcommand accepts `--mission`, `--actor`, `--dry-run`, and `--json` (on by default),
emits structured JSON on stdout, and uses stable error codes. Register the subgroup in
`cli/main.py` with a single added line. Cover all CLI-level behaviors with integration
tests.

## Context

- **spec.md FR-001, FR-002** define the subgroup surface and required flags.
- **spec.md FR-020** requires `--dry-run` on `open`/`resolve`/`defer`/`cancel`.
- **contracts/cli-contracts.md** is the authoritative CLI surface spec — implement
  exactly what is described there: argument names, JSON shapes, error codes.
- **contracts/decision_open_response.schema.json**,
  **contracts/decision_terminal_response.schema.json**, and
  **contracts/decision_verify_response.schema.json** define the output JSON shapes.
- The subgroup is registered under the `agent` Typer app in `cli/main.py`. Look at how
  existing subgroups (`charter`, `tasks`, `tests`, `release`) are registered and follow
  the same pattern — do NOT refactor or restructure `main.py`; only add the one new
  registration line.
- Exit codes: 0 success, 1 structured JSON error, 2+ unexpected exceptions.
- All errors output `{"error": "<msg>", "code": "<DECISION_CODE>", "details": {...}}`.

### T009 — Create decision.py (typer subgroup, 5 subcommands)

**Purpose.** Implement the `spec-kitty agent decision` typer subgroup with all five
subcommands, mapping CLI arguments to `service` and `verify` Python API calls, and
formatting all outputs as JSON.

**Steps.**
1. Create `src/specify_cli/cli/commands/decision.py`.
2. Define a `typer.Typer()` app named `decision_app` with `help="Decision Moment ledger for interview questions."`.
3. Implement a shared helper `_resolve_mission_dir(mission_handle: str) -> Tuple[Path, str, str]`
   that returns `(mission_dir, mission_id, mission_slug)`. Use the existing context resolver
   in `src/specify_cli/context/` or `src/specify_cli/paths.py` — inspect the existing
   `charter.py` or `implement.py` for the resolution pattern.
4. Implement a shared helper `_output(data: dict, json_mode: bool) -> None` that prints
   `json.dumps(data, indent=2)` if `json_mode`, else a plain text summary.
5. Implement a shared `_handle_decision_error(exc: DecisionError) -> NoReturn` that
   prints `{"error": ..., "code": ..., "details": ...}` to stdout and calls
   `raise typer.Exit(1)`.

**Subcommand: `open`**
```
@decision_app.command("open")
def cmd_open(
    mission: str = typer.Option(..., "--mission", help="Mission handle"),
    flow: str = typer.Option(..., "--flow", help="charter|specify|plan"),
    input_key: str = typer.Option(..., "--input-key"),
    question: str = typer.Option(..., "--question"),
    step_id: Optional[str] = typer.Option(None, "--step-id"),
    slot_key: Optional[str] = typer.Option(None, "--slot-key"),
    options: Optional[str] = typer.Option(None, "--options", help="JSON array string"),
    actor: str = typer.Option("cli", "--actor"),
    dry_run: bool = typer.Option(False, "--dry-run"),
    json_out: bool = typer.Option(True, "--json/--no-json"),
)
```
- Parse `options` JSON string → `List[str]` (default `[]` if None).
- Map `flow` → `OriginFlow` enum; reject invalid values with structured error.
- Call `service.open(...)`.
- On `DecisionError`: call `_handle_decision_error`.
- On success: output `DecisionOpenResponse` as JSON dict.

**Subcommand: `resolve`**
```
@decision_app.command("resolve")
def cmd_resolve(
    decision_id: str = typer.Argument(...),
    mission: str = typer.Option(..., "--mission"),
    final_answer: str = typer.Option(..., "--final-answer"),
    other_answer: bool = typer.Option(False, "--other-answer"),
    rationale: Optional[str] = typer.Option(None, "--rationale"),
    resolved_by: Optional[str] = typer.Option(None, "--resolved-by"),
    actor: str = typer.Option("cli", "--actor"),
    dry_run: bool = typer.Option(False, "--dry-run"),
    json_out: bool = typer.Option(True, "--json/--no-json"),
)
```
- Call `service.resolve(...)`.
- Output `DecisionTerminalResponse` as JSON dict.

**Subcommands: `defer` and `cancel`**
- Same pattern as `resolve` but call `service.defer` / `service.cancel`.
- `--rationale` is required (non-empty string); reject empty string with error
  `DECISION_MISSING_STEP_OR_SLOT` or a dedicated validation error.

**Subcommand: `verify`**
```
@decision_app.command("verify")
def cmd_verify(
    mission: str = typer.Option(..., "--mission"),
    fail_on_stale: bool = typer.Option(True, "--fail-on-stale/--no-fail-on-stale"),
    json_out: bool = typer.Option(True, "--json/--no-json"),
)
```
- Call `verify.verify(mission_dir, mission_slug)`.
- If `result.findings` is non-empty (and `fail_on_stale=True`): print JSON and
  `raise typer.Exit(1)`.
- If clean: print JSON and `raise typer.Exit(0)`.

### T020 — CLI integration tests: test_decision.py

**Purpose.** Verify CLI-level behavior: argument parsing, JSON output shapes, error
codes, exit codes, and `--dry-run` no-side-effects behavior.

**Steps.**
1. Create `tests/specify_cli/cli/commands/test_decision.py`.
2. Use `typer.testing.CliRunner` and a `tmp_path` fixture to create isolated mission
   directories. Patch `service.open`, `service.resolve`, etc. as needed to avoid
   real filesystem writes where testing CLI behavior only.
3. Write tests covering:
   a. `decision open --mission X --flow specify --slot-key foo --input-key bar --question "Q?" --dry-run`
      exits 0, outputs JSON with `decision_id="DRY_RUN"`.
   b. `decision open ...` (valid, non-dry-run) exits 0, JSON output validates against
      `decision_open_response.schema.json`.
   c. `decision open` without `--step-id` and without `--slot-key` exits 1 with
      `code="DECISION_MISSING_STEP_OR_SLOT"` in JSON.
   d. `decision open` with an already-closed decision (mock service raising
      `DecisionError(ALREADY_CLOSED)`) exits 1 with `code="DECISION_ALREADY_CLOSED"`.
   e. `decision resolve <id> --mission X --final-answer "Y"` exits 0, JSON validates
      against `decision_terminal_response.schema.json`.
   f. `decision resolve <id> --mission X --final-answer "Y" --other-answer` exits 0,
      response includes `terminal_outcome="resolved"`.
   g. `decision defer <id> --mission X --rationale "..."` exits 0, `terminal_outcome="deferred"`.
   h. `decision cancel <id> --mission X --rationale "..."` exits 0, `terminal_outcome="canceled"`.
   i. `decision verify --mission X` with a clean index exits 0, `status="clean"`.
   j. `decision verify --mission X` with a drift finding exits 1, `status="drift"`,
      `findings` array is non-empty.
   k. `decision verify --mission X --no-fail-on-stale` with a stale marker exits 0.
   l. Unknown `--flow badvalue` exits 1 with structured error.
