# WP06 — Charter integration

## Objective

Wire Decision Moment calls into the existing `spec-kitty charter interview` Q&A loop.
Before displaying each question, the charter command calls `service.open()` to mint a
`decision_id` and create the on-disk paper trail. After the user answers, it calls the
appropriate terminal command (`service.resolve()`, `service.defer()`, or
`service.cancel()`). The `answers.yaml` write logic is preserved exactly as-is; only
the Decision Moment overlay is added. This WP proves the coherent triple of Decision
Moment paper trail + events + `answers.yaml` all produced in a single charter run.

## Context

- **spec.md FR-012** requires charter to call `decision open` before each question and
  a terminal command after each answer, while preserving `answers.yaml` semantics.
- **spec.md scenario 1** (end-to-end charter flow) is the acceptance test for this WP.
- **spec.md C-005** explicitly preserves `answers.yaml` as primary charter state.
- **data-model.md §7** describes the charter integration: `origin_flow=CHARTER`,
  `step_id=f"charter.{question_id}"`.
- The charter interview loop lives in `src/specify_cli/cli/commands/charter.py`.
  Read the existing loop before editing to understand question iteration, `answers.yaml`
  write path, and the "skip" / "defer" UX (if any).
- `OriginFlow.CHARTER` is used for all charter-flow decisions.
- `answers.yaml` is written ONLY for resolved (non-deferred, non-canceled) answers
  (spec.md edge case, C-005).

### T011 — Wire decision calls into charter.py interview loop

2. Add imports at the top of `charter.py`:
   ```python

   from specify_cli.decisions.models import OriginFlow as DmOriginFlow
   from specify_cli.decisions.service import DecisionError
   ```
3. Inside the question-iteration loop, before displaying each question, add:
   ```python
   try:
       dm_response = dm_service.open(
           mission_dir=mission_dir,
           mission_id=mission_id,
           mission_slug=mission_slug,
           origin_flow=DmOriginFlow.CHARTER,
           step_id=f"charter.{question_id}",
           input_key=question_id,
           question=question_text,
           options=list(question_options) if question_options else [],
           actor=actor,
       )
       current_decision_id = dm_response.decision_id
   except DecisionError:
       current_decision_id = None  # non-fatal; charter continues
   ```
   Adjust the variable names to match the actual charter loop variables.
4. After capturing the user's answer, add the terminal call:
   - If the user's response is a real answer (to be written to `answers.yaml`):
     ```python

         try:
             dm_service.resolve(
                 mission_dir=mission_dir,
                 decision_id=current_decision_id,
                 final_answer=str(user_answer),
                 actor=actor,
             )
         except DecisionError:
             pass  # non-fatal
     ```

     ```python

             dm_service.defer(
                 mission_dir=mission_dir,

                 actor=actor,
             )
         except DecisionError:
             pass
     ```
   - If the user canceled / said "not applicable":
     ```python

         try:
             dm_service.cancel(
                 mission_dir=mission_dir,
                 decision_id=current_decision_id,
                 rationale="owner canceled during charter interview",
                 actor=actor,
             )
         except DecisionError:
             pass
     ```
5. Verify that the `answers.yaml` write only occurs on the resolved path (same as the
   existing logic — do NOT add a new write; ensure existing write is NOT moved or
   removed).
6. Determine how `actor` is available in the charter context; use the git config email
   or `"cli"` as fallback (consistent with `service.open()` default).

## Risks

- **Charter loop variability.** The charter loop may have multiple code paths for
  different question types (single-choice, multi-choice, free-text, yes/no). Ensure
  Decision Moment calls are added to ALL question-type paths, not just one.
- **actor resolution.** If charter does not currently resolve an `actor` string, add a
  minimal fallback: read `git config user.email` or default to `"cli"`.
- **Deferred/cancel UX detection.** The charter may have no explicit "skip" or "cancel"
  concept today. Identify how the user signals deferral (e.g., an empty answer, a
  special key, or a "skip" option in a menu) before writing the conditional. Do not
  silently map all non-answers to defer — preserve the existing charter UX exactly and
  only add Decision Moment side calls.
