# Feature Specification: Agent Skills Installer Infrastructure

**Feature Branch**: `042-agent-skills-installer-infrastructure`
**Created**: 2026-03-20
**Status**: Draft
**Input**: Phase 0 of the Agent Skills Pack v1 PRD — capability profile model, skill-pack install plumbing, managed manifest, and `spec-kitty init` distribution wiring.

## Context and Motivation

Spec Kitty currently distributes agent knowledge exclusively through duplicated command/prompt wrapper files generated per agent. Every supported agent gets a rendered copy of each mission template in its vendor-specific format (`.claude/commands/`, `.codex/prompts/`, `.gemini/commands/`, etc.).

As of 2026-03-20, 11 of 12 supported agents now expose native skill surfaces (`SKILL.md` bundles in discoverable project directories). Only Amazon Q lacks a documented first-class skill surface. Many agents support a shared `.agents/skills/` root, meaning one installed skill copy can serve multiple agents simultaneously.

The current `AGENT_COMMAND_CONFIG` in `src/specify_cli/core/config.py` only captures wrapper metadata (directory, extension, argument format). It has no concept of skill roots, distribution classes, or capability tiers. This blocks skill-native distribution entirely.

Phase 0 builds the foundational infrastructure to support skill-pack installation without authoring any skill content. Skill content authoring (1.x pack, 2.x pack) and distribution refinement are explicitly deferred to future phases.

### Source PRD

This spec implements **Phase 0: Capability Profile and Installer Refactor** from `prd-spec-kitty-agent-skills-pack-v1.md`. All design decisions and acceptance criteria trace to PRD sections 8.1–8.8 and Phase 0 in section 13.

## Assumptions

> These assumptions are current design positions. They can be revisited in future phases but do not block Phase 0.

- **A-001**: `.agents/skills/` is the shared default project skill root for agents that officially support it (Copilot, Gemini, Cursor, OpenCode, Windsurf, Codex, Augment, Roo).
- **A-002**: Vendor-native skill roots (`.claude/skills/`, `.qwen/skills/`, `.kilocode/skills/`) are used only for agents that require them (Claude, Qwen, Kilo Code).
- **A-003**: Amazon Q stays wrapper-only for Phase 0. No skill root is configured.
- **A-004**: Kiro is explicitly out of Phase 0 scope. It is not added to the supported agent list.
- **A-005**: Wrappers continue to be generated by default for all selected agents in Phase 0, even for highly skill-native agents. Making wrappers optional is a future-phase decision.
- **A-006**: Phase 0 does not author any skill content (no `SKILL.md` files). It only builds the machinery to install and manage them.
- **A-007**: The managed manifest tracks installed skill roots and wrapper hashes but does not enforce content integrity until skill packs exist (future phases).

## User Scenarios & Testing

### User Story 1 — Init with Skill-Aware Agent Distribution (Priority: P1)

A developer runs `spec-kitty init` to set up a new project and selects multiple agents. The system installs the correct skill roots based on each agent's distribution class, generates wrappers as before, and writes a managed manifest recording what was installed.

**Why this priority**: This is the core deliverable — without init producing skill roots, no future phase can install skill content.

**Independent Test**: Run `spec-kitty init --ai claude,codex,qwen` on a fresh directory and verify that `.agents/skills/` is created (shared root for Codex), `.claude/skills/` is created (native root for Claude), `.qwen/skills/` is created (native root for Qwen), existing wrappers are generated as before, and `.kittify/agent-surfaces/skills-manifest.yaml` is written with correct metadata.

**Acceptance Scenarios**:

1. **Given** a fresh project directory, **When** the user runs `spec-kitty init --ai claude,codex,opencode`, **Then** `.agents/skills/` is created (shared root for Codex and OpenCode), `.claude/skills/` is created (native root for Claude), all three agents receive their wrapper files as before, and a skills manifest is written.
2. **Given** a fresh project directory, **When** the user runs `spec-kitty init --ai amazonq`, **Then** no skill root directories are created, Amazon Q wrappers are generated as before in `.amazonq/prompts/`, and the manifest records Amazon Q as wrapper-only.
3. **Given** a fresh project directory, **When** the user runs `spec-kitty init --ai claude`, **Then** only `.claude/skills/` is created (no `.agents/skills/` because no shared-root agent was selected), wrappers are generated, and the manifest is written.

---

### User Story 2 — Richer Agent Capability Profile Model (Priority: P1)

A Spec Kitty developer needs to query an agent's distribution class, supported skill roots, and wrapper configuration from a single authoritative data structure. The new `AGENT_SURFACE_CONFIG` replaces `AGENT_COMMAND_CONFIG` and provides all capability metadata needed by init, migrations, sync, and future skill-pack operations.

**Why this priority**: Every other story depends on this data model existing. It is the single source of truth for agent capabilities.

**Independent Test**: Import `AGENT_SURFACE_CONFIG` from `src/specify_cli/core/config.py`, verify all 12 agents are present, verify each has distribution class, skill root precedence list, wrapper config, and compatibility notes.

**Acceptance Scenarios**:

1. **Given** `AGENT_SURFACE_CONFIG` is loaded, **When** querying the `codex` entry, **Then** it returns distribution class `shared-root-capable`, skill roots `[".agents/skills/"]`, and wrapper config `{dir: ".codex/prompts", ext: "prompt.md", arg_format: "$ARGUMENTS"}`.
2. **Given** `AGENT_SURFACE_CONFIG` is loaded, **When** querying the `claude` entry, **Then** it returns distribution class `native-root-required`, skill roots `[".claude/skills/"]`, and wrapper config preserved from current `AGENT_COMMAND_CONFIG`.
3. **Given** `AGENT_SURFACE_CONFIG` is loaded, **When** querying the `amazonq` entry, **Then** it returns distribution class `wrapper-only`, skill roots `[]`, and wrapper config preserved from current config.

---

### User Story 3 — Managed Install Manifest (Priority: P2)

After init completes, a YAML manifest at `.kittify/agent-surfaces/skills-manifest.yaml` records the Spec Kitty version, selected agents, installed skill roots, and managed wrapper files with content hashes. Future sync and upgrade operations can use this manifest to detect drift, repair missing files, and remove orphaned managed assets.

**Why this priority**: The manifest enables repair, sync, and upgrade safety for future phases. Without it, skill installation is fire-and-forget.

**Independent Test**: Run `spec-kitty init`, inspect the manifest file, verify it contains version, agent list, skill roots, and wrapper file entries with SHA-256 hashes.

**Acceptance Scenarios**:

1. **Given** init has completed for agents `[claude, codex, opencode]`, **When** reading the manifest, **Then** it contains entries for each installed skill root (`.agents/skills/`, `.claude/skills/`), each managed wrapper file, Spec Kitty version, and a timestamp.
2. **Given** an existing manifest from a previous init, **When** running init again with `--force`, **Then** the manifest is regenerated with current state.

---

### User Story 4 — Post-Init Verification (Priority: P2)

After installation completes, init runs a verification pass that checks every selected agent has either a managed skill root or a managed wrapper root, expected skill directories exist on disk, wrapper counts match the selected mission, and no duplicate skill names would be installed into multiple roots scanned by the same agent at the same precedence level.

**Why this priority**: Catches installation failures immediately rather than during later workflow execution.

**Independent Test**: Run init, then programmatically call the verification function and confirm it returns pass. Intentionally delete a skill root directory and confirm it returns a specific failure.

**Acceptance Scenarios**:

1. **Given** init completed successfully, **When** verification runs, **Then** all checks pass with no errors or warnings.
2. **Given** a shared-root-capable agent is selected but `.agents/skills/` was not created (simulated failure), **When** verification runs, **Then** it reports a specific error naming the agent and missing root.
3. **Given** a wrapper-only agent is selected, **When** verification runs, **Then** it checks for wrapper files only and does not expect skill roots.

---

### User Story 5 — Agent Config Sync Respects Skill Roots (Priority: P3)

Running `spec-kitty agent config sync` uses the manifest to detect missing managed skill roots or wrapper files and repairs them. It also removes orphaned managed assets for agents that have been removed from the config.

**Why this priority**: Extends the existing sync command to cover skills. Lower priority because Phase 0 installs empty skill roots (no content to repair yet), but the plumbing should be in place.

**Independent Test**: Run init, remove a skill root directory, run sync, verify it is recreated. Remove an agent from config, run sync, verify its managed skill root is cleaned up.

**Acceptance Scenarios**:

1. **Given** Claude is configured and `.claude/skills/` is missing, **When** running sync, **Then** the directory is recreated and the manifest is updated.
2. **Given** Codex was removed from config but `.agents/skills/` still exists and other agents use it, **When** running sync, **Then** the shared root is preserved (other agents still need it) but Codex-specific wrappers are removed.
3. **Given** an agent was removed and was the only user of a skill root, **When** running sync, **Then** the orphaned skill root is removed and manifest is updated.

---

### User Story 6 — Backward-Compatible Wrapper Generation (Priority: P1)

Existing projects that upgrade to the new version continue to generate wrappers exactly as before. The new `AGENT_SURFACE_CONFIG` preserves all wrapper metadata from the current `AGENT_COMMAND_CONFIG`. No wrapper file paths, extensions, or content formats change.

**Why this priority**: Breaking existing wrapper generation would be a regression for all current users.

**Independent Test**: Generate wrappers using current code, save output. Upgrade to new code, regenerate. Diff confirms identical output for all 12 agents.

**Acceptance Scenarios**:

1. **Given** an existing project with wrappers generated by the current version, **When** upgrading and running `spec-kitty upgrade`, **Then** all existing wrapper files remain unchanged.
2. **Given** the new `AGENT_SURFACE_CONFIG`, **When** the wrapper generator reads config for any agent, **Then** it produces identical wrapper content to the old `AGENT_COMMAND_CONFIG` path.

---

### Edge Cases

- What happens when a user runs init with `--ai` containing an unrecognized agent key? Rejected with an error listing valid keys (existing behavior, preserved).
- What happens when `.agents/skills/` already exists from a non-Spec-Kitty source? Init should not delete existing non-managed content. The manifest tracks only Spec-Kitty-managed files.
- What happens when two agents would install into the same native root? Should not occur given the current mapping (each native root maps to exactly one agent). If it did, verification would flag the duplicate.
- What happens during upgrade from a pre-Phase-0 project? A migration creates the manifest from current state and adds empty skill roots for configured agents.
- What happens when init is run with `--skills wrappers-only`? Only wrappers are generated, no skill roots, and the manifest records this mode. (This flag enables conservative teams to opt out of skill roots.)

## Requirements

### Functional Requirements

- **FR-001**: System MUST define an `AGENT_SURFACE_CONFIG` data structure containing, per agent: distribution class (`shared-root-capable`, `native-root-required`, `wrapper-only`), skill root paths in precedence order, wrapper config (directory, extension, argument format), and compatibility notes.
  Status: Draft

- **FR-002**: `AGENT_SURFACE_CONFIG` MUST cover all 12 currently supported agents with accurate distribution class assignments per PRD section 8.1.
  Status: Draft

- **FR-003**: System MUST preserve all existing wrapper metadata from `AGENT_COMMAND_CONFIG` within `AGENT_SURFACE_CONFIG` so that wrapper generation produces identical output.
  Status: Draft

- **FR-004**: `spec-kitty init` MUST resolve required skill roots from the selected agent set and create the minimum set of project skill root directories.
  Status: Draft

- **FR-005**: For shared-root-capable agents, init MUST install a shared project skill root at `.agents/skills/` when at least one such agent is selected.
  Status: Draft

- **FR-006**: For native-root-required agents, init MUST install vendor-specific skill roots (`.claude/skills/`, `.qwen/skills/`, `.kilocode/skills/`) only for selected agents of that class.
  Status: Draft

- **FR-007**: For wrapper-only agents, init MUST NOT create any skill root directories. Wrapper generation continues unchanged.
  Status: Draft

- **FR-008**: `spec-kitty init` MUST accept a `--skills` flag with values `auto`, `native`, `shared`, `wrappers-only`. Default is `auto`.
  Status: Draft

- **FR-009**: When `--skills auto`, init MUST install `.agents/skills/` when any shared-root-capable agent is selected, install native-only roots where required, and continue generating wrappers for all selected agents.
  Status: Draft

- **FR-010**: When `--skills wrappers-only`, init MUST skip all skill root creation and generate only wrappers (preserving pre-Phase-0 behavior).
  Status: Draft

- **FR-011**: When `--skills native`, init MUST prefer vendor-native skill roots even for agents that also support `.agents/skills/`.
  Status: Draft

- **FR-012**: When `--skills shared`, init MUST prefer `.agents/skills/` wherever official docs allow, falling back to native-only roots for agents that require them.
  Status: Draft

- **FR-013**: Init MUST write a managed install manifest to `.kittify/agent-surfaces/skills-manifest.yaml` recording: Spec Kitty version, selected agents, installed skill roots, managed wrapper files with SHA-256 content hashes, installation mode, and timestamp.
  Status: Draft

- **FR-014**: Init MUST run a post-install verification pass confirming: every selected agent has a managed skill root or wrapper root, expected directories exist, wrapper counts match the mission, and no duplicate skill names exist in overlapping roots.
  Status: Draft

- **FR-015**: Verification failures MUST be reported as actionable error messages naming the specific agent and missing resource. Verification MUST NOT silently pass.
  Status: Draft

- **FR-016**: `spec-kitty agent config sync` MUST read the manifest and repair missing managed skill roots and wrapper files for configured agents.
  Status: Draft

- **FR-017**: `spec-kitty agent config sync` MUST remove orphaned managed skill roots for agents no longer in the config, but MUST preserve shared roots still needed by other configured agents.
  Status: Draft

- **FR-018**: System MUST provide a `get_agent_surface(agent_key)` function that returns the full capability profile for a given agent key, replacing direct `AGENT_COMMAND_CONFIG` lookups in calling code.
  Status: Draft

- **FR-019**: The `asset_generator.py` wrapper generation path MUST be refactored to read wrapper config from `AGENT_SURFACE_CONFIG` instead of `AGENT_COMMAND_CONFIG`, with identical output.
  Status: Draft

- **FR-020**: A migration MUST be provided that upgrades pre-Phase-0 projects by: creating the manifest from current state, adding empty skill roots for configured agents based on their distribution class, and preserving all existing wrapper files.
  Status: Draft

- **FR-021**: The migration MUST be config-aware (using `get_agent_dirs_for_project()`) and MUST NOT create skill roots for agents not in the project's config.
  Status: Draft

- **FR-022**: Init MUST generate skill roots into the project directory as empty managed directories with a `.gitkeep` or equivalent marker so they survive version control.
  Status: Draft

- **FR-023**: The manifest MUST distinguish between Spec-Kitty-managed files and user-authored content to prevent sync or repair from deleting non-managed files in shared directories.
  Status: Draft

### Non-Functional Requirements

- **NFR-001**: `AGENT_SURFACE_CONFIG` MUST be defined as a typed Python data structure (dataclass or TypedDict) with mypy strict compliance.
  Status: Draft

- **NFR-002**: Init skill-root resolution MUST complete in under 100ms for any combination of agents (it is pure data lookup, no I/O beyond directory creation).
  Status: Draft

- **NFR-003**: New code MUST maintain 90%+ test coverage per project constitution.
  Status: Draft

- **NFR-004**: All new functions MUST pass `mypy --strict` with no type errors.
  Status: Draft

- **NFR-005**: All new functions MUST pass `ruff check` with no lint errors.
  Status: Draft

- **NFR-006**: Migration MUST be idempotent — running it multiple times on the same project produces the same result.
  Status: Draft

### Constraints

- **C-001**: Phase 0 MUST NOT author any skill content (`SKILL.md` files). Only installation machinery is in scope.
  Status: Active

- **C-002**: Phase 0 MUST NOT add Kiro to the supported agent list. Kiro support is future scope.
  Status: Active

- **C-003**: Phase 0 MUST NOT remove or make wrappers optional. Wrappers are generated for all selected agents as before.
  Status: Active

- **C-004**: Phase 0 MUST NOT install user-global skills. Only project-local skill roots are in scope.
  Status: Active

- **C-005**: Phase 0 MUST NOT add a `--track` flag to init. Track selection (1.x vs 2.x) is deferred to Phase 1/2 when skill packs exist.
  Status: Active

- **C-006**: `AGENT_COMMAND_CONFIG` may be retained temporarily as an alias or migration aid, but `AGENT_SURFACE_CONFIG` MUST be the authoritative structure for new code.
  Status: Active

- **C-007**: No changes to the mission system, template rendering, or command-template content are in scope.
  Status: Active

### Key Entities

- **AgentSurfaceConfig**: Per-agent capability profile containing distribution class, skill roots, wrapper config, and compatibility notes. Replaces `AGENT_COMMAND_CONFIG` as the authoritative agent metadata.

- **DistributionClass**: Enum with values `shared-root-capable`, `native-root-required`, `wrapper-only`. Determines how init resolves skill roots for an agent.

- **SkillsManifest**: Persistent YAML structure recording installed skill roots, managed files with content hashes, selected agents, installation mode, Spec Kitty version, and timestamps. Stored at `.kittify/agent-surfaces/skills-manifest.yaml`.

- **SkillRoot**: A project directory where skill bundles can be installed. Either shared (`.agents/skills/`) or vendor-native (`.claude/skills/`, etc.). Phase 0 creates these as empty managed directories.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Running `spec-kitty init` with any combination of agents produces the correct minimum set of skill root directories as defined by the distribution class matrix — verifiable by automated test across all 12 agents.

- **SC-002**: Running `spec-kitty init` followed by diffing wrapper output against pre-Phase-0 wrapper output produces zero differences for all 12 agents — backward compatibility is byte-exact.

- **SC-003**: The managed manifest accurately reflects installed state — verifiable by programmatic comparison of manifest entries against filesystem contents.

- **SC-004**: Post-init verification catches 100% of simulated installation failures (missing roots, missing wrappers) in automated tests.

- **SC-005**: `spec-kitty agent config sync` repairs missing managed skill roots and removes orphaned roots — verifiable by delete-and-repair test cycle.

- **SC-006**: The upgrade migration successfully converts pre-Phase-0 projects without data loss — verifiable by running the migration on fixture projects and confirming wrapper files are preserved and manifest is created.

- **SC-007**: New code achieves 90%+ test coverage and passes `mypy --strict` and `ruff check` with zero errors.
