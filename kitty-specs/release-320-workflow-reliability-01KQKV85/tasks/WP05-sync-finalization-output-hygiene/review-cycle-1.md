**Issue 1**: Text-surface local success still falls back to unstructured final-sync warnings instead of the required non-fatal sync diagnostic.

The WP05 contract applies after a local state mutation is durable, not only after a JSON command explicitly calls `mark_invocation_succeeded()`. In `src/specify_cli/sync/background.py`, the new structured diagnostic path is gated by `invocation_succeeded()` at the lock, shutdown, timeout, and `_guarded_final_sync()` exception branches. When that flag is false, the code emits logger warnings such as `Final sync failed: network down` and `Could not acquire sync lock within 5 s; skipping final sync` without the required `severity`, `diagnostic_code`, `fatal=false`, `sync_phase`, and `message` fields.

I verified this with a mocked final-sync failure after printing a text success line and without calling `mark_invocation_succeeded()`: stdout kept the local success text, but stderr was only `Final sync failed: network down`. That violates the contract's `text_commands` rule and T027/T028 diagnostic-shape requirement.

Fix by making post-local-success final-sync failures use the structured `SyncDiagnostic` emission path for text surfaces as well, or by passing explicit local-mutation success state into finalization instead of relying only on the JSON success flag. Add a focused regression test that exercises final-sync failure after a text command/local success without `mark_invocation_succeeded()` and asserts the stderr diagnostic includes the required fields and no red failure prefix.
