## R1 — Inline-reference rejection: skip vs propagate (drives FR-001, FR-003, I-1/I-9)

**Context.** `repository.py:423-424` deliberately *propagates* `InlineReferenceRejectedError` ("a hard, fail-closed contract violation … so the author fixes the error"). `diagnostics.py:6` lists inline-ref rejection as a *skip reason*. The doctor health collector `_collect_profile_health` wraps profile loading in a broad `except Exception` that degrades to an empty report → `healthy = all([]) = True`. Net: a pack with an inline-ref-invalid profile is reported HEALTHY and all profiles vanish (the #1584 false-healthy class, reproduced).

**Decision (REVISED after planning review — A1/DD-1):** Catch `InlineReferenceRejectedError` in `repository._load_layer` and route it through `_record_skip` (recording a `SkippedProfile{layer, path, profile_id, error_summary}`), so it becomes a **recoverable skip** — the invalid profile is surfaced as skipped while its **valid siblings still load**, and the per-file catch has the YAML in hand to populate the `profile_id`. `_collect_profile_health` then reports `healthy=False` (non-empty invalid/skipped set) without any change to its broad-except. Reconcile `diagnostics.py`'s docstring and the `repository.py` comment to agree that inline-ref rejection is a **surfaced skip** (resolving I-9 in lockstep — this aligns the code to the docstring that already calls it a skip).

**Rationale (why the original "surgical consumer-only" default was rejected):** Profile loading is **eager and all-or-nothing** — `repository._load()`/`list_all()` raises mid-iteration, aborting the entire layer load, so a consumer-side catch in `_collect_profile_health` can mark the pack unhealthy but **cannot** satisfy contract C1's "other valid profiles remain visible" (reviewers alphonso A1 + debbie DD-1 reproduced a mixed 2-pack case where one inline-ref profile blanked a valid sibling in another pack *and* the built-in layer). Additionally, the exception does **not** carry `profile_id` (DD-2), which the consumer seam cannot recover without re-reading the YAML — but the load-layer skip already has it. The load-layer fix is therefore the **only** design that satisfies C1 and supplies the full `{path, id, error_summary}` shape.

**Blast radius (accepted, with regression coverage):** Making inline-ref rejection a skip means general callers (`resolve_profile`/`get_ancestors`) no longer raise on it — a broken profile becomes *unavailable* (skipped) rather than crashing. This matches the existing `diagnostics.py` contract and the #1584 intent ("surface invalid profiles, don't hide/crash"). NFR-001 regression coverage (incl. `tests/doctrine/agent_profiles/` and `tests/doctrine/test_inline_ref_rejection.py`) MUST confirm no general-caller regression; if any caller genuinely needs the hard-fail, it asserts on `skipped_profiles()` instead.

## R2 — FR-036 dead-symbol claim accuracy (drives FR-009, I-6)

**Decision:** Remove the two redundant re-exports (`SignificanceEvaluatedPayload`, `TimeoutExpiredPayload`) from `events.py.__all__` — the canonical definitions and live callers are in `…_internal_runtime/significance.py`; the `events.py` entries are unused re-exports masking that fact. ## R3 — Cascade-warning removal (drives FR-008, I-5)

**Decision (REVISED after planning review — A2/P-2/DD-3):** **Delete the stale warning blocks** in `pack_manager`. Passing `cascade=False` was proven WRONG by three reviewers: `pack_manager` has **two** deferral-warning branches — `if cascade:` (~lines 424-429, "not yet implemented … deferred") **and** `if not cascade:` (~lines 417-423, "re-run with --cascade … cascade execution deferred"). So `cascade=False` merely swaps to the *other* "deferred" message (and mislabels a real cascade as non-cascade), still failing the contract. The real cascade is CLI-owned via `charter.cascade` / `_render_cascade_activation`; `pack_manager`'s `cascade` parameter does **no real work** (debbie verified `commit_plan` is identical regardless), so it is vestigial. Remove **both** activate branches (~417-429) and the deactivate equivalent (~493-498). Add a test asserting "not yet implemented"/"deferred" is absent from both `charter activate --cascade` and `charter deactivate --cascade` output.
**Rationale:** Only deleting the stale branches removes the forbidden substring on *all* paths. Safe because the branches are pure warnings with no behavioral effect.
**Test-update obligation (DD-4):** the existing `test_activate_cascade_calls_with_true` (asserts `cascade=True` at the call site) and `test_activate_cascade_flag_accepted` (asserts a "Warning"/"cascade" string present) MUST be updated to the new contract; no other test pins the deferral strings. These belong in the FR-008 task.
