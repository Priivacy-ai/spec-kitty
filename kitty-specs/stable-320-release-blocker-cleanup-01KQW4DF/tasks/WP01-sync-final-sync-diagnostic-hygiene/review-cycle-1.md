**Issue 1**: Auth-refresh lock contention is still swallowed before final-sync retry/diagnostic handling.

The WP explicitly calls out the live smoke signal `Another spec-kitty process is refreshing the auth session; retry in a moment` and requires it to route through the final-sync diagnostic authority as `sync.auth_refresh_in_progress` after bounded retries. In the current implementation, the real shutdown final-sync path does not reach `run_final_sync_with_retries()` with that error.

The problem is in `src/specify_cli/sync/background.py`: `_fetch_access_token_sync()` catches all unexpected exceptions at lines 113-118 and returns `None`. `RefreshLockTimeoutError`, which carries the exact "refreshing the auth session" message, is defined in `src/specify_cli/auth/refresh_transaction.py` as a plain `Exception`, not an `AuthenticationError`. Then `_sync_once()` treats `None` at lines 357-361 as a no-op unauthenticated skip and returns an empty `BatchSyncResult`. Because `run_final_sync_with_retries()` only retries or emits when an exception escapes or `error_count > 0`, this live auth-refresh failure gets zero retries and zero `sync.auth_refresh_in_progress` diagnostic.

Fix by preserving this condition as a transient final-sync failure that reaches the retry/classification layer. For example, import and handle `RefreshLockTimeoutError` explicitly in `background.py`, either by letting it propagate from `_fetch_access_token_sync()` during final sync or by returning a `BatchSyncResult` with `error_count > 0` and the original message. Add a regression test that exercises `BackgroundSyncService.stop()` or `_guarded_final_sync()` with the real refresh-lock timeout path and asserts three attempts/backoff plus exactly one stderr diagnostic with `diagnostic_code=sync.auth_refresh_in_progress` and clean stdout.
