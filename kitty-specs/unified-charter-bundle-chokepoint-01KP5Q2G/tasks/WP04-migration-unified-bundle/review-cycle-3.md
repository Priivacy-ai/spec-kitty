**Issue 1**: `detect()` is too narrow for the current upgrade runner, so the migration is skipped on stale/no-charter 3.2.2 -> 3.2.3 upgrades instead of running and producing the required no-op or refresh behavior. `MigrationRunner._apply_migration()` hard-skips whenever `detect()` is false ([src/specify_cli/upgrade/runner.py](/Users/robert/spec-kitty-dev/spec-kitty-charter-14-April/spec-kitty/.worktrees/unified-charter-bundle-chokepoint-01KP5Q2G-lane-a/src/specify_cli/upgrade/runner.py:193)), but `UnifiedBundleMigration.detect()` only returns true when derived files are missing ([m_3_2_3_unified_bundle.py](/Users/robert/spec-kitty-dev/spec-kitty-charter-14-April/spec-kitty/.worktrees/unified-charter-bundle-chokepoint-01KP5Q2G-lane-a/src/specify_cli/upgrade/migrations/m_3_2_3_unified_bundle.py:103)). That means fixture (d) stale metadata and fixture (e) no-charter pass only via direct `migration.apply()`, not via the real upgrade path. I reproduced this with a temp 3.2.2 repo: `MigrationRunner(...).upgrade('3.2.3')` skipped `m_3_2_3_unified_bundle` entirely on a stale bundle and still stamped metadata to `3.2.3`. Fix by aligning the migration with runner semantics: either broaden `detect()` so every 3.2.2 -> 3.2.3 project that should emit a report is considered applicable, or change the runner contract used by this migration so `apply()` runs and decides between refresh vs clean no-op.

**Issue 2**: The schema-shaped migration report is not emitted through the documented `spec-kitty upgrade --json` surface. The WP and schema both state that `m_3_2_3_unified_bundle` should emit the structured JSON report on `upgrade --json`, but the CLI currently discards per-migration payloads and only returns summary status fields ([src/specify_cli/cli/commands/upgrade.py](/Users/robert/spec-kitty-dev/spec-kitty-charter-14-April/spec-kitty/.worktrees/unified-charter-bundle-chokepoint-01KP5Q2G-lane-a/src/specify_cli/cli/commands/upgrade.py:322)). The new tests only validate `json.loads(result.changes_made[0])` from direct `migration.apply()` ([test_unified_bundle_migration.py](/Users/robert/spec-kitty-dev/spec-kitty-charter-14-April/spec-kitty/.worktrees/unified-charter-bundle-chokepoint-01KP5Q2G-lane-a/tests/upgrade/test_unified_bundle_migration.py:198)), so they miss the operator-visible contract. Fix by plumbing the migration report through the actual upgrade JSON output and add an integration test that exercises the upgrade command/runner path rather than only the migration class directly.
