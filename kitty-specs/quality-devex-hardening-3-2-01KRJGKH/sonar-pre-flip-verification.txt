OPTION A — Defer T038 flip until a follow-up WP raises coverage.

  Scope: A new WP (or WP05 patch) must add tests for the low-coverage files before
  the flip. Estimated effort: significant — the files are large (charter.py: 891
  lines, engine.py: 502 lines) and currently at 25-40% coverage. To lift the
  aggregate above 80%, approximately 7,000+ new covered lines are needed.

  Risk: Coverage work may introduce test-environment dependencies or require
  architectural changes to make the CLI commands testable in isolation.

  Recommended if: The operator wants full gate-green before enabling push Sonar.

OPTION B — Renegotiate Sonar threshold (tiered or lowered).

  Approach: In SonarCloud project settings, reduce new_coverage threshold
  temporarily (e.g., to 60% matching the actual new-code coverage), or configure
  per-file exclusions for untestable surfaces (dashboard.js, generated code).

  The research artifact (work/findings/595-sonar-coverage-debt.md) documents
  Pedro's prior recommendation against resetting the new-code baseline. Tiering
  (different thresholds per file type) is the middle path.

  Risk: Lowering the gate makes future coverage regressions invisible. The gate
  loses its teeth as a quality signal.

  Recommended if: The operator accepts a temporary threshold relaxation and plans
  to tighten it incrementally.

OPTION C — Reset the new-code baseline.

  Approach: In SonarCloud, set a new baseline date so the new-code window starts
  from the current commit. Sonar re-computes new_coverage against only code added
  after that date, which would show a different (likely higher) coverage percentage
  for fresh additions.

  Risk (Pedro's note from research §2): Resetting the baseline hides accumulated
  coverage debt. Subsequent PRs will appear green even if large debt remains in
  "old code." This is the operator's decision to make.

  Recommended if: The operator concludes the current debt is an acceptable baseline
  and wants a clean slate for new contributions.
