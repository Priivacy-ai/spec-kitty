# WP03 introduces canonical terms `pipeline-shape` and `rule pipeline`

```yaml
terms:
  - surface: pipeline-shape
    definition: |
      A classification label applied to a function whose body is a sequence of independent
      decisions over a shared input. Pipeline-shape functions are the principal refactor target
      for the `chain-of-responsibility-rule-pipeline` tactic. The shape marker is "3+ independent
      decisions over a common input" — not "function is long". Distinct from deliberate linearity,
      where the sequential structure is intentional and documented. See
      `architecture/2.x/04_implementation_mapping/code-patterns.md` entry 1.
    confidence: 0.9
    status: active

  - surface: rule pipeline
    definition: |
      A pipeline of small pure functions, each checking applicability, optionally executing its
      narrow piece of work, and returning a result threaded to the next function. Three flavors are
      recognized in this codebase: Validator (accumulating `list[Finding]`), Transformer (chained
      `(state, actions, error?)`), Scorer (weighted `float`). The Transformer flavor is implemented
      in `src/specify_cli/migration/canonicalization.py::CanonicalRule` and consumed by
      `mission_state.py::_canonicalize_status_row` and `rebuild_state.py`. See the
      `chain-of-responsibility-rule-pipeline` tactic.
    confidence: 0.95
    status: active
```
