# WP04 introduces canonical term `catastrophic backtracking`

```yaml
terms:
  - surface: catastrophic backtracking
    definition: |
      A regular-expression engine failure mode in which a vulnerable pattern combined with
      adversarial input drives matching time into exponential or polynomial complexity.
      Backtracking engines (Python `re`, JavaScript, Java, .NET) attempt every possible
      partition of the input when a match fails; patterns with nested or consecutive
      non-possessive repetitions can produce O(2^N) or O(N^c) retry storms. The
      `secure-regex-catastrophic-backtracking` tactic at
      `src/doctrine/tactics/shipped/secure-regex-catastrophic-backtracking.tactic.yaml`
      codifies the four dangerous-shape patterns, the rewrite ladder (bound quantifier ->
      refactor nested -> possessive/atomic group -> negated char class), and the escape
      hatches (drop the regex, re2, multi-pass, bounded-input cap). Every fix requires a
      wall-clock regression test asserting linear runtime on an adversarial input
      (default budget: <= 100 ms for 100 000 chars), per FR-008. Synonym "ReDoS"
      (Regular-expression Denial of Service) is acceptable only as a parenthetical.
    confidence: 0.95
    status: active
    see_also:
      - tactic: secure-regex-catastrophic-backtracking
        path: src/doctrine/tactics/shipped/secure-regex-catastrophic-backtracking.tactic.yaml
      - fr: FR-008
        description: Wall-clock regression test requirement for every regex change
```
