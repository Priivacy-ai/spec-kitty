# Governance Files Reference

The Charter governance layer lives in `.kittify/charter/`. Most files are auto-generated by the
synthesizer and must not be hand-edited. This page is the authoritative reference for every file
in that directory.

> **Key rule**: `charter.md` is the only human-edited governance file. Everything else is
> overwritten on the next `charter synthesize` or `charter sync` run. Editing generated files
> creates drift that `charter status` and `charter lint` will detect.

---

## File Table

| File path | Who writes it | Contains | Edit directly? |
|---|---|---|---|
| `.kittify/charter/charter.md` | **Human** (via interview or direct edit) | Mission vision, directives, doctrine source, policy decisions | Yes — this is the only human-edited file |
| `.kittify/charter/governance.yaml` | Auto-generated (synthesis) | Resolved directives in structured form | No — overwritten on next synthesis |
| `.kittify/charter/directives.yaml` | Auto-generated (synthesis) | Extracted directive list with severity and scope | No — overwritten on next synthesis |
| `.kittify/charter/metadata.yaml` | Auto-generated (synthesis) | Bundle metadata: charter hash, last-sync timestamp, provenance | No — overwritten on next synthesis |
| `.kittify/charter/library/*.md` | Auto-generated (`charter generate`) | Doctrine library pages copied from packaged doctrine set | No — overwritten on next generate |
| `.kittify/charter/interview/answers.yaml` | Auto-generated (`charter interview`) | Captured interview answers used as input to `charter generate` | No — re-run `charter interview` to update |
| `.kittify/doctrine/` | Auto-generated (`charter synthesize`) | Project-local doctrine promoted by synthesizer | No — overwritten on next synthesize |
| `.kittify/doctrine/PROVENANCE.md` | Auto-generated (`charter synthesize`) | Human-readable provenance record for the synthesis run | No |

## What Happens If You Edit a Generated File

The synthesizer can overwrite generated-file edits on the next `charter synthesize` run. If you
edit `governance.yaml` or `directives.yaml` directly and then run `charter synthesize`, your edits
will be silently lost.

Use these commands to detect drift before relying on the bundle:

```bash
# Check whether charter.md is out of sync with the bundle
uv run spec-kitty charter status

# Detect orphaned artifacts, contradictions, and staleness in the graph
uv run spec-kitty charter lint

# Validate the bundle against the canonical schema
uv run spec-kitty charter bundle validate
```

If `charter status` reports drift, re-run the synthesis flow from `charter synthesize` to
regenerate clean outputs from the current `charter.md`.

---

## Bundle Validation

The charter bundle is validated against the **CharterBundleManifest v1.0.0** schema by:

```bash
uv run spec-kitty charter bundle validate
```

This verifies that all required files are present, correctly structured, and consistent with
each other. Run it after every synthesis to confirm a valid bundle before running missions.

`charter lint` performs graph-native decay checks — it detects orphaned directives (directives
that appear in the DRG but have no referencing tactic), contradictions (two directives with
conflicting instructions), and staleness (a directive whose provenance references a deleted or
superseded shipped directive).

---

## Sync vs Synthesize

These two operations are different:

| Command | What it does |
|---|---|
| `charter sync` | Syncs `charter.md` content to YAML config files (governance.yaml, directives.yaml, metadata.yaml). Use after hand-editing charter.md. |
| `charter synthesize` | Validates and promotes agent-generated project-local doctrine artifacts from `.kittify/charter/generated/` to `.kittify/doctrine/`. This is the full DRG-backed synthesis step. |

Run `charter sync` first when you have edited `charter.md` by hand, then `charter synthesize`
to promote the updated doctrine.
