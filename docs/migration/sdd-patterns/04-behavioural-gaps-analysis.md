# Behavioural Gaps Analysis

_Migration research document — gaps between current SDD Patterns agentic framework and
spec-kitty doctrine model_
_Date: 2025-05-25_

## 1. Executive Summary

This document identifies the behavioural and structural gaps between the existing
penguin-pragmatic-patterns agentic framework and the spec-kitty doctrine model. Each gap
is classified by severity, with proposed remediation paths for the doctrine bundle migration.

## 2. Gap Classification

| Severity | Meaning |
|----------|--------|
| **Critical** | Blocks pack creation or renders artifacts unusable |
| **High** | Degrades functionality or violates spec-kitty invariants |
| **Medium** | Missing feature that limits effectiveness but has workarounds |
| **Low** | Enhancement opportunity; current state is functional |

## 3. Structural Gaps

### GAP-01: No Machine-Readable Artifact Format (Critical)

**Current state:** Agent profiles, directives, guidelines, and style rules are authored as
freeform Markdown files (`.github/agents/*.agent.md`, `.github/agents/directives/*.md`,
`.github/agents/guidelines/*.md`).

**Spec-kitty requires:** YAML artifacts with `schema_version`, `id`, `title`, `description`,
`action_scope`, `enforcement`, and other structured fields. Each artifact kind has a
validated schema (`src/doctrine/schemas/`).

**Impact:** Cannot be loaded by `DoctrineService`, cannot be validated by `spec-kitty doctor
doctrine`, cannot participate in DRG queries.

**Remediation:**
1. Convert each Markdown agent file to `*.agent.yaml`
2. Convert each directive to `*.directive.yaml`
3. Convert guidelines to `*.styleguide.yaml` or `*.directive.yaml`
4. Validate against spec-kitty schemas

**Effort:** High (40+ files to convert)

---

### GAP-02: No DRG Graph Definition (Critical)

**Current state:** No graph definition exists. Relationships between agent profiles,
directives, style rules, and content types are implicit in the Markdown text.

**Spec-kitty requires:** Explicit `drg/fragment.yaml` files with `nodes` and `edges`
declaring URN-based relationships (e.g., `source: "urn:directive:sdd-001"`,
`target: "urn:action:implement"`, `relation: scope`).

**Impact:** Cannot participate in context-scoped selection. Charter compiler cannot resolve
which artifacts apply to which mission actions.

**Remediation:**
1. Create `drg/010-sdd-content-types.graph.yaml`
2. Create `drg/020-sdd-curation-workflow.graph.yaml`
3. Create `drg/030-sdd-quality-gates.graph.yaml`
4. Validate with `spec-kitty doctrine drg validate`

**Effort:** Medium (design-heavy, implementation straightforward)

---

### GAP-03: No Pack Manifest (Critical)

**Current state:** No `pack-manifest.yaml` exists. Artifacts are not inventoried or
versioned as a distributable unit.

**Spec-kitty requires:** A manifest generated by `spec-kitty doctrine pack assemble`
that tracks pack version, build timestamp, artifact inventory, and DRG integrity hash.

**Impact:** Cannot be distributed or consumed by other projects.

**Remediation:**
1. Organize artifacts into pack directory structure
2. Run `spec-kitty doctrine pack assemble` (once available)
3. Or manually create manifest following schema

**Effort:** Low (automated by tooling once artifacts exist)

---

### GAP-04: No Org-Charter Policy File (High)

**Current state:** No `org-charter.yaml` exists. Required and suggested doctrine selections
are not declared at the organization level.

**Spec-kitty requires:** `org-charter.yaml` with `required_*` and `suggested_*` fields
that propagate to consumer projects via `apply_org_charter_to_interview`.

**Impact:** Consumer projects cannot inherit SDD governance defaults.

**Remediation:**
1. Create `org-charter.yaml` declaring required directives, styleguides, and paradigms
2. Add activation registry entries for context-scoped selections
3. Validate against org charter schema

**Effort:** Low-Medium

---

### GAP-05: Glossary Format Mismatch (High)

**Current state:** Glossary is in TOML format (`data/glossary.toml`) with fields: `name`,
`abbreviation`, `domain`, `description`, `aliases`, `references`.

**Spec-kitty requires:** YAML format with fields: `surface` (canonical display form),
`definition`, `context`, `status` (candidate/canonical/deprecated/superseded),
`confidence` (0.0-1.0), `see_also`, `scope` (spec_kitty_core/team_domain/audience_domain/
mission_local).

**Impact:** Glossary cannot be loaded by spec-kitty's glossary semantic pipeline. Terms
cannot participate in DRG, conflict detection, or strictness enforcement.

**Remediation:**
1. Write a TOML-to-YAML conversion script
2. Map fields: `name` → `surface`, `description` → `definition`, `domain` → `context`
3. Set initial `status: canonical` and `confidence: 0.9` for all terms
4. Set `scope: team_domain` for SDD-specific terms
5. Generate `see_also` from `aliases` and `references`

**Effort:** Medium (scripted conversion + manual review)

---

## 4. Governance Gaps

### GAP-06: No Charter Interview Integration (Medium)

**Current state:** Agent initialization uses a fixed 6-layer context stack loaded in
sequence. There is no interactive interview that adapts governance to project needs.

**Spec-kitty provides:** A Charter Interview that walks the HiC through preferences and
constraints, producing `answers.yaml` that the Charter Compiler uses to build the charter.

**Impact:** Every project using SDD Patterns gets the same governance, with no ability to
customize selections or override defaults.

**Remediation:**
1. Define interview questions specific to SDD Patterns context
   - "Which content types will this project produce?" (practice/concept/primer)
   - "What citation format do you prefer?" (IEEE/APA/none)
   - "What tone strictness level?" (relaxed/standard/strict)
2. Map answers to directive activation/deactivation
3. Include in org-charter.yaml as interview defaults

**Effort:** Medium

---

### GAP-07: No Status Tracking for Content Lifecycle (Medium)

**Current state:** Content status is tracked implicitly through Git branches and PR state.
There is no formal lane/status model for pattern curation.

**Spec-kitty provides:** A 9-lane state machine (planned → claimed → in_progress →
for_review → in_review → approved → done) with append-only event logs, deterministic
reducers, and transition guards.

**Impact:** No programmatic visibility into curation pipeline state. Cannot auto-detect
which patterns are in progress, blocked, or ready for review.

**Remediation:**
1. Adopt spec-kitty's status model for pattern curation missions
2. Use `documentation` mission type with adapted workflow sequence
3. Map curation stages to lane transitions (see Journey Map document)

**Effort:** Low (adopt existing spec-kitty infrastructure)

---

### GAP-08: No Activation Context Mapping (Medium)

**Current state:** Directives are loaded via `/require-directive NNN` commands. There is
no automatic activation based on mission type or action.

**Spec-kitty provides:** Activation Registry with `(activation_context, doctrine_pack_id,
artifact_id)` tuples that automatically inject context for specific `mission_type × action`
combinations.

**Impact:** Agents must manually load directives. Risk of forgetting relevant directives
for specific actions.

**Remediation:**
1. Define activation contexts for each SDD directive
2. Register in org-charter.yaml activation_registry
3. Map SDD content types to spec-kitty mission actions

**Effort:** Low-Medium

---

### GAP-09: No Precedence Conflict Resolution (Low)

**Current state:** Conflict handling is defined in AGENTS.md as
"Operational > Strategic > Command convenience" with manual halt-and-flag.

**Spec-kitty provides:** Formal precedence rule:
`CLI override > step metadata > mission config > global default`
with machine-enforced resolution.

**Impact:** Conflict resolution relies on agent judgment rather than deterministic rules.

**Remediation:**
1. Map SDD precedence layers to spec-kitty's precedence model
2. Bootstrap Protocol → global default
3. Operational Guidelines → mission config
4. Project Specific → step metadata
5. Command aliases → CLI override

**Effort:** Low

---

## 5. Content Gaps

### GAP-10: AMMERSE Evaluation Has No Doctrine Equivalent (Medium)

**Current state:** Every practice pattern includes an AMMERSE evaluation with 7 dimensions
(agile, minimal, maintainable, environmental, reachable, solvable, extensible), each rated
with a delta value and rationale.

**Spec-kitty lacks:** No built-in artifact kind or template field for multi-dimensional
evaluation frameworks.

**Impact:** AMMERSE evaluation cannot be declared as a doctrine requirement. It exists
only as a template convention, not as a validated schema.

**Remediation options:**
1. **Directive approach:** Create `sdd-007-ammerse-delta-required` directive requiring
   AMMERSE fields in content frontmatter
2. **Template Set approach:** Include AMMERSE as a required section in the practice
   template set
3. **Schema extension approach:** Propose upstream spec-kitty schema extension for
   evaluation frameworks (long-term)

**Effort:** Low (directive + template set covers the requirement)

---

### GAP-11: Bibliography System Has No Doctrine Equivalent (Low)

**Current state:** Bibliography is managed in `data/bibliography.toml` with auto-generated
Hugo content pages and cross-linking via `further_exploration` frontmatter.

**Spec-kitty lacks:** No built-in artifact kind for reference/bibliography management.

**Impact:** Bibliography management remains project-local. Cannot be shared across
projects via doctrine pack.

**Remediation options:**
1. **Toolguide approach:** Document bibliography workflow as a toolguide artifact
2. **Data artifact approach:** Include `bibliography.toml` as a pack data file
   (non-doctrine, project-level)
3. **Keep project-local:** Bibliography is inherently project-specific and may not
   need doctrine-level treatment

**Recommendation:** Keep bibliography project-local. Document the workflow as a toolguide.

---

### GAP-12: Contextive IDE Integration Not Modeled (Low)

**Current state:** `.contextive/*.glossary.yml` files provide IDE hover definitions and
auto-complete for terminology. They are auto-generated from `glossary.toml`.

**Spec-kitty lacks:** No built-in support for IDE glossary integration. Spec-kitty's
glossary pipeline focuses on semantic integrity and conflict detection, not IDE tooltips.

**Impact:** IDE integration remains a project-level convention.

**Remediation:**
1. Document Contextive integration as a toolguide
2. Include generation script reference in tactic
3. Propose upstream consideration for IDE glossary rendering (long-term)

---

### GAP-13: Reasoning Mode Protocol Not Modeled (Low)

**Current state:** Three reasoning modes (analysis, creative, meta) with explicit
transition annotations (`[mode: creative → analysis]`).

**Spec-kitty lacks:** No built-in concept of agent reasoning modes. Agent profiles define
roles and governance scope but not cognitive operating modes.

**Impact:** Mode-switching behavior stays as freeform agent guidance rather than
machine-validated doctrine.

**Remediation options:**
1. **Paradigm approach:** Model each mode as a paradigm (analysis, creative, meta)
2. **Procedure approach:** Model mode transitions as a procedure
3. **Keep as-is:** Reasoning modes are agent-internal behavior and may not need
   doctrine-level modeling

**Recommendation:** Model as paradigms with a mode-switching procedure.

---

### GAP-14: Integrity Symbol System Not Modeled (Low)

**Current state:** Three integrity symbols (❗️ critical, ⚠️ low confidence, ✅ confirmed)
used in agent output to signal alignment status.

**Spec-kitty equivalent:** Partial — the clarification burst policy and glossary strictness
policy cover similar territory but at the governance level, not at the output formatting
level.

**Impact:** Purely cosmetic. Agents can adopt this convention without doctrine enforcement.

**Remediation:** Include as a styleguide rule in `sdd-tone-map.styleguide.yaml`.

---

## 6. Tooling Gaps

### GAP-15: No spec-kitty doctrine pack CLI Support Yet (Medium)

**Current state:** Spec-kitty's `doctrine pack` CLI subcommand exists in design documents
and glossary definitions but may not be fully implemented for third-party pack creation.

**Impact:** Pack assembly, validation, and distribution may require manual steps until
the CLI matures.

**Remediation:**
1. Monitor spec-kitty releases for `doctrine pack assemble` and `doctrine pack validate`
2. Use manual pack creation in the interim (directory structure + manifest)
3. Contribute pack tooling improvements upstream if needed

---

### GAP-16: No TOML-to-YAML Glossary Converter (Medium)

**Current state:** No tooling exists to convert the penguin-pragmatic-patterns TOML
glossary format to spec-kitty's YAML glossary format.

**Impact:** Glossary migration requires custom scripting.

**Remediation:**
1. Write a Python conversion script
2. Include in pack build pipeline
3. Maintain bidirectional sync during transition period

## 8. Recommended Remediation Order

### Phase 1: Pack Foundation (Critical gaps)

1. **GAP-01** — Convert Markdown artifacts to YAML schema
2. **GAP-05** — Convert TOML glossary to YAML format
3. **GAP-02** — Create DRG fragment files
4. **GAP-03** — Generate pack manifest
5. **GAP-04** — Create org-charter.yaml

### Phase 2: Governance Integration (High/Medium gaps)

6. **GAP-08** — Define activation contexts
7. **GAP-06** — Design charter interview questions
8. **GAP-07** — Adopt status model for curation

### Phase 3: Content Enrichment (Medium/Low gaps)

9. **GAP-10** — Model AMMERSE as directive + template set
10. **GAP-09** — Map precedence to spec-kitty model
11. **GAP-13** — Model reasoning modes as paradigms
12. **GAP-16** — Build glossary conversion tooling

### Phase 4: Enhancement (Low gaps)

13. **GAP-11** — Document bibliography as toolguide
14. **GAP-12** — Document Contextive as toolguide
15. **GAP-14** — Include integrity symbols in styleguide
